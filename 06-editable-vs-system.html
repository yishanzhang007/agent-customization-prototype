Editable vs system nodes
Action vs scaffolding

Signaling what's editable

On the canvas, some nodes are clinic-editable (intents, actions) and some are platform scaffolding (incoming call, classify intent). Today they all look identical, so clicking around to discover what's clickable is the only way to learn. Below are six UX directions for making the distinction visual.

Editable: Welcome message, intent cards, action cards. System: Incoming call (the start), Classify intent (the router). Both kinds always render — they just respond differently to interaction.

Baseline (today)

Identical treatment — no visual cue distinguishes the two.

${nodeSystemBaseline('Incoming call', 'Business hours · Mon–Fri 9–5')} ${nodeSystemBaseline('Classify intent', '5 intents')} ${nodeEditableBaseline('Take details', '→ Clinical team')}
Option A

Filled vs ghost

strong contrast

System nodes lose their fill — outline only, transparent background. They blend with the canvas and read as "structure."

${nodeGhost('Incoming call', 'Business hours · Mon–Fri 9–5')} ${nodeGhost('Classify intent', '5 intents')} ${nodeEditable('Take details', '→ Clinical team')}

+ unmissable distinction · − ghost cards can feel "disabled" or "incomplete"

Option B

Hover affordance only

subtle

Identical at rest. Hover the editable nodes → they lift + show pencil + cursor:pointer. System nodes have cursor:default and don't react.

${nodeStatic('Incoming call', 'Business hours · Mon–Fri 9–5')} ${nodeStatic('Classify intent', '5 intents')} ${nodeHoverPencil('Take details', '→ Clinical team')}

+ no visual noise at rest · − discoverability is poor — you have to hover to learn

Option C

Dashed border for system

peripheral

System nodes get a dashed border, mirroring the "+ Add intent" empty-state pattern already used to indicate "this is structural, not your content."

${nodeDashed('Incoming call', 'Business hours · Mon–Fri 9–5')} ${nodeDashed('Classify intent', '5 intents')} ${nodeEditable('Take details', '→ Clinical team')}

+ subtle, reuses existing dashed-border vocabulary · − conflicts with the dashed "add intent" button

Option D

Glass / translucent

ambient

System nodes are semi-transparent with a slight blur — like a watermark. Editable cards "pop" against them as the real content.

${nodeGlass('Incoming call', 'Business hours · Mon–Fri 9–5')} ${nodeGlass('Classify intent', '5 intents')} ${nodeEditable('Take details', '→ Clinical team')}

+ feels modern, layered · − contrast issues on textured backgrounds

Option E

Different shape

strong distinction

System nodes become pills/chips — smaller, rounder, single-line. Editable nodes stay as cards. Shape = role.

${nodePill('Incoming call · Business hours')} ${nodePill('Classify intent · 5')} ${nodeEditable('Take details', '→ Clinical team')}

+ truly different shape signals truly different role · − requires re-laying out mode info that the start node currently shows

Option F

Lock icon on system

explicit

Small 🔒 (or "Managed") icon top-right on system nodes. Reads like file-permissions UI — instantly understood as "you can't touch this."

${nodeLocked('Incoming call', 'Business hours · Mon–Fri 9–5')} ${nodeLocked('Classify intent', '5 intents')} ${nodeEditable('Take details', '→ Clinical team')}

+ semantically clear · − adds a tiny icon that competes for attention with the override badge / custom indicators

Recommended combo

Glass (D) + Hover affordance (B)

System nodes go translucent (D) — at rest, they read as scaffolding without screaming "disabled." Editable nodes keep the white card + lift on hover (B), so the interactive zone is unmistakable when you mouse over. Cursor stays default on system nodes; pointer on editable. Together: clear at-a-glance role, intuitive on interaction.

${nodeGlass('Incoming call', 'Business hours · Mon–Fri 9–5')} ${nodeGlass('Classify intent', '5 intents')} ${nodeHoverPencil('Take details', '→ Clinical team')}

Try mousing over the cards above.

Things to decide
  • Welcome message is currently editable (you can change the opening line). Treat it as editable, not system.
  • Classify node has some configurable bits (classifier mode, fallback). It's a borderline case — could be treated as system (immutable structure) or as editable with a "settings" affordance.
  • Cursor matters more than visual. Whatever direction we pick, `cursor: default` on system nodes will already prevent the click-and-see-nothing-happens frustration.
  • Hover affordance is necessary regardless — even with strong rest-state differentiation, hovering an editable card should still confirm "yes you can click me" via lift + cursor change.
